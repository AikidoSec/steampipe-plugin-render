# AGENTS.md — render-steampipe

A Steampipe plugin that exposes the Render API as queryable SQL tables. Modeled on
[turbot/steampipe-plugin-supabase](https://github.com/turbot/steampipe-plugin-supabase).

## Goal

Let users query Render resources (services, deploys, postgres, etc.) with SQL via Steampipe.
Not all of the Render API is covered yet — see the [Roadmap](#roadmap) section.

## Layout

```
render-steampipe/
├── main.go                              # entry point: plugin.Serve()
├── render/
│   ├── plugin.go                        # plugin definition + table registration
│   ├── connection_config.go             # api_key / api_url HCL config
│   ├── service.go                       # client factory (memoized per connection)
│   ├── table_render_<name>.go           # one file per table (17 of them)
│   └── client/                          # generated from public-api-schema (do NOT edit by hand)
│       ├── client_gen.go                # the typed HTTP client
│       ├── types_gen.go                 # request/response types
│       └── {autoscaling,blueprints,disks,events,jobs,…}/  # subpackage types
├── config/render.spc                    # connection config template (copy to ~/.steampipe/config/)
├── docs/                                # markdown docs per table
├── generate.sh                          # regenerate render/client/ from the schema
├── oapi-render.yaml                     # oapi-codegen config
├── Makefile
├── go.mod
└── README.md
```

## How the plugin talks to Render

`render/client/` is generated by [oapi-codegen](https://github.com/oapi-codegen/oapi-codegen) from
[`renderinc/public-api-schema`](https://github.com/renderinc/public-api-schema). The generated
client gives us a typed `ClientWithResponses` with `ListXxxWithResponse` / `RetrieveXxxWithResponse`
methods for every endpoint.

To regenerate after a schema change:

```sh
go install github.com/oapi-codegen/oapi-codegen/v2/cmd/oapi-codegen@latest
PUBLIC_API_SCHEMA_PATH=~/Desktop/projects/public-api-schema ./generate.sh
```

`generate.sh` defaults `PUBLIC_API_SCHEMA_PATH` to `../public-api-schema` if unset.

## Build / install / test loop

```sh
make install                  # rebuild + copy to ~/.steampipe/plugins/hub.steampipe.io/plugins/render-oss/render@latest/
steampipe service stop --force
steampipe query "select id, name from render_service"   # auto-starts service with new plugin
```

`steampipe service restart` doesn't exist — use `stop --force` followed by any `query` invocation.
The connection config is at `~/.steampipe/config/render.spc`. Auth is via either `api_key` in that
file or the `RENDER_API_KEY` env var (the steampipe service must be started from a shell that has
the env var).

When debugging a failed query, the most useful logs are:
- `~/.steampipe/logs/plugin-YYYY-MM-DD.log` — plugin-side errors and Render API call URLs.
- `~/.steampipe/logs/steampipe-YYYY-MM-DD.log` — Postgres FDW side.

## Dependency pins

A few `go.mod` entries are deliberately held back. `steampipe-plugin-sdk/v5` (current pin
v5.13.1; same problem persists through v5.14.x and v6.0.0 as of 2026-06-19) is written
against the v1 line of `dgraph-io/ristretto` and the v0.0.x line of `olekukonko/tablewriter`.
Newer majors break the SDK at compile time:

- `ristretto/v2` added a generic `Key` constraint that rejects `interface{}` — breaks
  `connection/connection_cache.go`.
- `tablewriter v1.x` removed `SetHeader` / `SetBorder` / `SetColWidth` / `AppendBulk` /
  `SetAutoWrapText` — breaks `logging/timing.go`.

For that reason these stay pinned:

- `github.com/turbot/steampipe-plugin-sdk/v5` — at v5.13.1
- `github.com/eko/gocache/store/ristretto/v4` — at v4.2.1 (v4.3.x switched to ristretto/v2)
- `github.com/olekukonko/tablewriter` — at v0.0.5

`golang.org/x/*` security bumps are fine and should continue to flow through; the
`fix(security)` autofix bot has already done several. The hazard is automated bumps of the
three packages above — if one slips through, `make build` will fail with errors inside the
SDK, not in this repo's code (that's the tell). Re-check upstream the next time we touch
the SDK version: if the SDK has migrated to `ristretto/v2` and `tablewriter v1.x`, all three
pins can be lifted at once.

There's a matching `KEEP THESE PINNED` block at the top of `go.mod`.

## Tables

17 tables, organized in tiers as the project grew:

**Tier 1 (foundation):**
- `render_owner` — workspaces (top-level)
- `render_project` — projects (top-level)
- `render_environment` — sub-resource of `render_project` (parent hydrate)
- `render_service` — services (top-level, the `render_service` you'll join most)
- `render_deploy` — sub-resource of `render_service` (parent hydrate)

**Tier 2 (other top-level resources):**
- `render_postgres` — Postgres databases
- `render_key_value` — Key-value (Redis-compatible) instances
- `render_disk` — persistent disks (parent-hydrated by `render_owner` for `owner_id` enrichment)
- `render_env_group` — env var groups; `env_vars`/`secret_files` are hydrated lazily
- `render_blueprint` — render.yaml IaC; `resources` hydrated lazily; parent-hydrated by `render_owner`
- `render_registry_credential` — container registry creds; parent-hydrated by `render_owner`

**Tier 3 (per-service sub-resources, all parent-hydrated by `render_service`):**
- `render_custom_domain`
- `render_secret_file` — names only, content **deliberately omitted** even though the API returns it
- `render_header`
- `render_route`
- `render_notification_override` — top-level (not parent-hydrated by service); parent-hydrated by `render_owner` for `owner_id` enrichment
- `render_job`

## Patterns / conventions

### Naming
- **Match the Render API exactly.** The API uses `owner` for workspaces; we keep `render_owner` and
  `owner_id` rather than renaming to `workspace` (matches docs and external terminology). Decision:
  consistency with the API beats matching the dashboard's customer-facing language.
- Each table file is `render/table_render_<name>.go` and exposes one `tableRender<Name>(ctx)` factory
  function plus list/get hydrates.

### Cursor pagination
Every list endpoint returns `[]ItemWithCursor` where each item has its own cursor field. To paginate:
1. Send `?limit=100` (default page size).
2. Stream items, remembering the **last** item's cursor.
3. If the page returned fewer than `limit` items, stop. Otherwise, send another request with
   `?cursor=<last-item-cursor>&limit=100`.

Some endpoints return a flat `[]Item` with no per-item cursor — in those cases we make a single
request and stop. Affected: `render_env_group`, `render_registry_credential`. Listing >100 of these
will under-report; revisit if it bites.

### 404-on-cursor = end of pages
The Render API sometimes returns 404 when paginating past the last page (rather than an empty
array). Every paginating list hydrate handles this defensively:

```go
if resp.JSON200 == nil {
    if resp.StatusCode() == 404 && params.Cursor != nil {
        return nil, nil  // end of pages
    }
    return nil, fmt.Errorf("list X failed: ...")
}
```

The `params.Cursor != nil` guard means a 404 on the **first** request still surfaces as a real
error.

### Parent-hydrated tables
For sub-resources (e.g. `render_deploy` sub of `render_service`), we use Steampipe's `ParentHydrate`:

```go
List: &plugin.ListConfig{
    ParentHydrate: listRenderServices,
    Hydrate:       listRenderDeploys,
    KeyColumns:    plugin.OptionalColumns([]string{"service_id", ...}),
},
```

**The child hydrate receives the parent via `h.Item`, NOT `h.ParentItem`.** This is non-obvious
from the field names — `HydrateData` has both fields, but the SDK only populates `Item` when
calling a parent-hydrated child list. From `steampipe-plugin-sdk/v5/plugin/query_data.go:616`:

```go
_, err := d.Table.List.Hydrate(ctx, childQueryData, &HydrateData{Item: parentItem})
```

Use:

```go
service := h.Item.(client.Service)
```

(I burned a full debug session writing `h.ParentItem.(client.Service)` everywhere. It's silently
nil, which after a "defensive nil-check" returned empty rows for every parent-hydrated table
without erroring. Don't do this.)

If the user provides a key qual (e.g. `where service_id = 'srv-X'`), the child hydrate still runs
once per parent; we skip non-matching parents inline:

```go
if v := d.EqualsQualString("service_id"); v != "" && v != service.Id {
    return nil, nil
}
```

This avoids wasted API calls per non-matching parent.

### owner_id enrichment via parent hydrate
Some Render API list responses (`/blueprints`, `/registry-credentials`, `/disks`,
`/notification-overrides`) accept `ownerId` as a **filter** but don't echo it back in each row.
Earlier we tried surfacing `owner_id` as a `transform.FromQual("owner_id")` filter-only column,
but that returned NULL when the user didn't add a `WHERE owner_id = …`.

The current pattern is to parent-hydrate by `render_owner`, iterate one API call per workspace, and
wrap the response in a struct that carries the owner ID:

```go
type Blueprint struct {
    blueprints.Blueprint
    OwnerId string
}
```

Trade-off: `1 + N` API calls where N is the number of workspaces the API key can see. For a
single-workspace key (most common) that's 2 calls instead of 1 — negligible.

### Secret files
`render_secret_file.content` is **deliberately not exposed** even though the API returns it. The
table only lists names/paths. Querying secret values out of a SQL prompt is too easy a footgun —
people who need contents can use the API directly.

### Filter-only columns vs lazy hydrate
- **Filter-only column** (`transform.FromQual("col")`) — useful when the API accepts a filter but
  the response doesn't include the value. Simple but the column is NULL unless the user filters.
  We've moved away from this in favor of parent hydrate (see above).
- **Lazy hydrate column** (`Hydrate: getX, Transform: FromField(...)`) — fires a per-row Retrieve
  call only when that column is in the SELECT. Used for `render_blueprint.resources` and
  `render_env_group.env_vars` / `secret_files`.

## Common pitfalls

- **Key column declared but column not defined** → plugin fails to start with "key column 'X' does
  not exist in table 'Y'". Every key column needs a matching `Column` entry. (Bit me on
  `render_disk.owner_id`.)
- **Parent hydrate panicking with `interface {} is nil`** → add the `h == nil || h.ParentItem ==
  nil` guard.
- **Steampipe still showing old plugin behavior after `make install`** → run
  `steampipe service stop --force` (NOT `restart`, which doesn't exist).
- **Stale Postgres backend after a session crash** → `ps aux | grep postgres.*\.steampipe`,
  `kill <pid>`, then re-query.
- **Plugin starts but queries error with "api_key must be configured"** → either the
  `~/.steampipe/config/render.spc` `api_key` line is commented out, or the `steampipe service` was
  started from a shell that doesn't export `RENDER_API_KEY`.
- **`plugin = "render"` doesn't resolve** → must be `plugin = "render-oss/render"` to match the
  install path.

## Roadmap

Tiers shipped: 1, 2, 3. Remaining work in priority order:

**Tier 4 (governance + observability):**
- `render_event` — service event log (deploy_started, suspended, server_failed, …). Sub-resource of
  service. Probably the highest-value remaining table.
- `render_webhook` — outbound webhook configs.
- `render_log_stream` — log-stream destinations (top-level).
- `render_maintenance` — maintenance windows for managed services.
- `render_snapshot` — disk snapshots, sub-resource of `render_disk`.
- `render_blueprint_sync` — blueprint sync history, sub-resource of `render_blueprint`.
- `render_postgres_export` — Postgres logical exports, sub-resource of `render_postgres`.
- `render_redis` — legacy Redis. Render is migrating customers off this; only add if needed.

**Skipped (poor SQL fit):**
- `ListLogs` / `ListLogsValues` — log streaming, time-series, not relational.
- `ListPathFilterValues` / `ListHttpFilterValues` / `ListApplicationFilterValues` — log filter
  helpers (static enums).
- `ListWebhookEvents` — list of available webhook event types (a static enum, not a queryable
  resource).
